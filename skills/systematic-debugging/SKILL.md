# 系統化除錯（Systematic Debugging）

## 概觀

隨機亂修浪費時間，還會製造新 bug。快速補丁會遮蓋底層問題。

**核心原則：**一律（ALWAYS）在嘗試修正之前先找出根本原因。修症狀就是失敗。

**違反這個流程的字面，就是違反除錯的精神。**

## 鐵律（The Iron Law）

```
沒有先做根本原因調查，就不得進行任何修正
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果你還沒完成 Phase 1，你就不能提出修正。

## 何時使用

用於任何（ANY）技術問題：
- 測試失敗
- 正式環境的 bug
- 非預期行為
- 效能問題
- 建置（build）失敗
- 整合問題

**尤其（ESPECIALLY）在以下情況使用：**
- 時間壓力下（緊急時很容易想用猜的）
- 「只要一個快速修正」看似顯而易見時
- 你已經試過多個修正
- 前一個修正沒效
- 你並未完全理解這個問題

**不要在以下情況略過：**
- 問題看似簡單（簡單的 bug 也有根本原因）
- 你在趕時間（趕只會保證返工）
- 主管要你「現在馬上」修好（系統化比亂試更快）

## 四個 Phase

你必須（MUST）先完成每個 phase，才能進到下一個。

### Phase 1：根本原因調查

**在嘗試任何（ANY）修正之前：**

1. **仔細閱讀錯誤訊息**
   - 不要略過錯誤或警告
   - 它們常常就包含確切的解法
   - 完整讀完 stack trace
   - 記下行號、檔案路徑、錯誤碼

2. **穩定重現**
   - 你能可靠地觸發它嗎？
   - 確切的步驟是什麼？
   - 每次都發生嗎？
   - 若無法重現 → 蒐集更多資料，不要用猜的

3. **檢查近期變更**
   - 什麼變動可能造成這個？
   - Git diff、近期 commit
   - 新的相依套件、config 變更
   - 環境差異

4. **在多元件系統中蒐集證據**

   **當系統有多個元件時（CI → build → signing、API → service → database）：**

   **在提出修正之前，加上診斷用的 instrumentation：**
   ```
   對每個 component 邊界:
     - 記錄進入 component 的資料
     - 記錄離開 component 的資料
     - 驗證 environment／config 的傳遞
     - 檢查每一層的狀態

   先跑一次,蒐集能顯示「在哪裡壞掉」的證據
   接著分析證據,找出出問題的 component
   接著只調查那個特定的 component
   ```

   **範例（多層系統）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **這會揭露：**哪一層失敗（secrets → workflow ✓、workflow → build ✗）

5. **追蹤資料流**

   **當錯誤深藏在 call stack 裡時：**

   完整的反向追蹤技巧見本目錄的 `root-cause-tracing.md`。

   **快速版：**
   - 壞掉的值從哪裡來？
   - 是什麼用這個壞值呼叫了它？
   - 一路往上追，直到找到源頭
   - 在源頭修，不要在症狀處修

### Phase 2：模式分析

**修之前先找出模式：**

1. **找出可運作的範例**
   - 在同一個程式碼庫裡找相似、可運作的程式碼
   - 有什麼跟壞掉的東西相似、卻是好的？

2. **對照參考實作**
   - 若在實作某個模式，完整（COMPLETELY）讀過參考實作
   - 不要略讀——每一行都讀
   - 在套用之前完全理解該模式

3. **辨識差異**
   - 可運作的與壞掉的之間有什麼不同？
   - 列出每一個差異，無論多小
   - 不要假設「那不可能有影響」

4. **理解相依關係**
   - 這需要哪些其他元件？
   - 需要哪些設定、config、環境？
   - 它做了哪些假設？

### Phase 3：假設與測試

**科學方法：**

1. **形成單一假設**
   - 清楚陳述：「我認為 X 是根本原因，因為 Y」
   - 把它寫下來
   - 要具體，不要含糊

2. **做最小的測試**
   - 做「最小可能」的變更來測試假設
   - 一次一個變數
   - 不要一次修好幾件事

3. **繼續之前先驗證**
   - 有效嗎？有 → Phase 4
   - 沒效？形成新的假設
   - 不要在上面再疊更多修正

4. **當你不知道時**
   - 說「我不理解 X」
   - 不要假裝知道
   - 尋求協助
   - 再多做研究

### Phase 4：實作

**修根本原因，不是修症狀：**

1. **建立會失敗的測試案例**
   - 最簡單可能的重現
   - 可以的話用自動化測試
   - 沒有框架就寫一次性的測試腳本
   - 修正前必須（MUST）先有
   - 用 `superpowers:test-driven-development` skill 來寫正確的失敗測試

2. **實作單一修正**
   - 針對辨識出的根本原因
   - 一次一個（ONE）變更
   - 不要「順手」改進
   - 不要夾帶重構（refactoring）

3. **驗證修正**
   - 現在測試通過了嗎？
   - 沒有弄壞其他測試嗎？
   - 問題真的解決了嗎？

4. **若修正沒效**
   - 停（STOP）
   - 數一數：你已經試了幾個修正？
   - 若 < 3：回到 Phase 1，用新資訊重新分析
   - **若 ≥ 3：停下並質疑架構（見下方第 5 點）**
   - 未經架構討論，不要嘗試第 4 個修正

5. **若 3 個以上修正都失敗：質疑架構**

   **顯示架構問題的徵兆：**
   - 每個修正都在不同的地方揭露新的共享狀態／耦合／問題
   - 修正需要「大規模重構」才能實作
   - 每個修正都在別處製造新症狀

   **停下並質疑根本：**
   - 這個模式本質上健全嗎？
   - 我們是不是「純粹靠慣性硬撐」？
   - 我們該重構架構，還是繼續修症狀？

   **在嘗試更多修正之前，與你合作的人類使用者討論**

   這不是失敗的假設——這是錯誤的架構。

## 警訊（Red Flags）——停下並遵循流程

如果你發現自己在想：
- 「先快速修一下，晚點再調查」
- 「就改改 X，看看有沒有用」
- 「加一堆變更，跑測試」
- 「跳過測試，我手動驗證」
- 「大概是 X 吧，我來修那個」
- 「我沒完全理解，但這樣可能有用」
- 「模式說 X，但我會改個方式套用」
- 「主要問題是：[沒調查就列出一堆修正]」
- 在追蹤資料流之前就提出解法
- **「再試一個修正」（明明已經試過 2 個以上）**
- **每個修正都在不同的地方揭露新問題**

**以上全部都代表：停。回到 Phase 1。**

**若 3 個以上修正都失敗：**質疑架構（見 Phase 4.5）

## 你合作的人類使用者發出的「你做錯了」訊號

**留意這些糾正：**
- 「那個沒有發生嗎？」——你沒驗證就假設了
- 「這會不會讓我們看到……？」——你早該加上證據蒐集
- 「別再猜了」——你在沒理解的情況下提修正
- 「深入想清楚這個（Ultra-think this）」——質疑根本，而不只是症狀
- 「我們卡住了？」（帶著挫折）——你的方法沒用

**當你看到這些時：**停。回到 Phase 1。

## 常見合理化藉口

| 藉口 | 事實 |
|--------|---------|
| 「問題很簡單，不用走流程」 | 簡單的問題也有根本原因。流程對簡單 bug 很快。 |
| 「緊急，沒時間走流程」 | 系統化除錯比亂猜亂試更快。 |
| 「先試這個，再調查」 | 第一個修正就定了調。從一開始就做對。 |
| 「等確認修正有效再寫測試」 | 沒測過的修正撐不住。測試先行才能證明。 |
| 「一次多個修正省時間」 | 無法隔離是哪個有效。會製造新 bug。 |
| 「參考太長，我改一下模式套用」 | 一知半解保證出 bug。完整讀完。 |
| 「我看到問題了，讓我修」 | 看到症狀 ≠ 理解根本原因。 |
| 「再試一個修正」（在 2 次以上失敗後） | 3 次以上失敗 = 架構問題。質疑模式，別再修。 |

## 快速參考

| Phase | 關鍵活動 | 成功標準 |
|-------|---------------|------------------|
| **1. 根本原因** | 讀錯誤、重現、查變更、蒐證 | 理解「是什麼」與「為什麼」 |
| **2. 模式** | 找可運作的範例、對照 | 辨識差異 |
| **3. 假設** | 形成理論、最小測試 | 確認，或形成新假設 |
| **4. 實作** | 建立測試、修正、驗證 | Bug 解決、測試通過 |

## 當流程顯示「沒有根本原因」

若系統化調查顯示問題確實是環境性的、與 timing 相關，或屬外部因素：

1. 你已經完成流程
2. 記錄你調查了什麼
3. 實作適當的處理（retry、timeout、錯誤訊息）
4. 加上 monitoring／logging 供日後調查

**但是：**95% 的「沒有根本原因」案例其實是調查不完整。

## 真實世界的影響

來自多次除錯 session:
- 系統化方法：15–30 分鐘修好
- 隨機亂修方法：2–3 小時的瞎折騰
- 一次修好率：95% 對 40%
- 引入的新 bug：近乎零 對 常見
